14.1 本章问题

你有结构规格包（Part 3）和构造件选择（Part 4）。但结构规格不是代码，构造件选择不是实现。下一步是为每类结构规格确定实现方式：标记自写代码部分、构造件承接部分和脚手架生成部分，并补齐连接它们的胶水代码。

14.2 背景：为什么这章重要

程序组装阶段承接需求场景、结构规格包和构造件选择表，并进一步拆成实现方案、胶水代码连接和最小可运行系统三个动作。

实现方案采用三分法：自写代码、使用构造件、生成脚手架。每一个结构规格都需要这个决策。

旧模型的实现方案是”结构→代码”的直接映射——对象卡片映射到 class，动作规格映射到函数。新模型的实现方案多出了两个维度： 1. 构造件承接：大部分结构不需要自己写代码，构造件已经实现好了 2. 胶水代码：构造件不是插上就能用，需要适配代码连接

这一章做的是：为每一个结构规格写上实现决策表。

14.3 核心内容

14.3.1 实现方案三分法

自写代码（Write）：业务逻辑的核心代码，具有业务特异性，没有现成构造件可以承接。

使用构造件（Use）：通用的程序结构，用库、框架、组件、脚手架承接。

生成脚手架（Scaffold）：项目壳和基础目录结构。

以下代码用于展示结构如何落地，不代表完整生产配置。

实现方式 适用场景 例子 风险
自写代码 业务逻辑、领域约束、胶水代码 状态转移验证、权限检查、API 适配 重复造轮子
使用构造件 通用结构 Web 框架、ORM、UI 组件库 过度依赖、升级风险

# 自写代码：领域结构中的生命周期约束
# 没有现成构造件承接状态转移逻辑，需要自己写
def validate_transition(current: str, target: str) -> bool:
    allowed = {
        "open": ["in_progress", "closed"],
        "in_progress": ["pending_review", "failed"],
        "pending_review": ["done", "in_progress"],
        "done": ["closed"],
        "closed": [],
        "failed": ["in_progress"],
    }
    if target not in allowed.get(current, []):
        raise StateError(f"不能从 {current} 转移到 {target}")

14.4 例子与实践

14.4.1 场景：Task Manager 的实现方案

# task_service.py — 自写代码（核心业务逻辑）
class TaskService:
    def __init__(self, repo: TaskRepository, notifier: NotificationService):
        self.repo = repo
        self.notifier = notifier

    def create(self, title: str, description: str | None,
               priority: str, creator_id: int,
               due_date: str | None = None) -> Task:
        # 自写代码：领域约束
        if not title or len(title) > 200:
            raise ValidationError("title 1-200 characters")

        # 构造件：SQLAlchemy 承接数据持久化
        task = self.repo.create(
            title=title,
            description=description,
            priority=priority,
            status="open",
            creator_id=creator_id,
            due_date=due_date,
        )

        # 自写代码：触发后续动作
        self.notifier.notify_creator(task)
        return task

    def assign(self, task_id: int, assignee_id: int) -> Task:
        task = self.repo.get(task_id)
        if not task:
            raise NotFoundError("任务不存在")

        # 自写代码：状态转移验证
        if task.status != "open":
            raise StateError("只能分配待处理任务")

        # 自写代码 + 构造件
        task.assignee_id = assignee_id
        task.status = "in_progress"
        updated = self.repo.update(task)

        self.notifier.notify_assignee(updated)
        return updated

14.5 小结

从结构模型到实现方案的三分法：自写代码、使用构造件、生成脚手架。

每一类结构规格都需要这个决策。自写代码处理业务特异性构造件不处理的部分，构造件处理通用结构，脚手架处理项目壳。

实现方案的产物是一张实现决策表——标出每个结构元素用什么方式实现、用什么构造件、需要什么胶水代码。

有了实现方案，下一章可以进入代码级别：自写代码和构造件怎么通过胶水代码连接。

14.7 从业者输出

完成本章后，你能做三件事：

第一，对一个结构规格包写出完整的实现决策表。你能为每个结构元素标记”自写/构造件/脚手架”。

第二，识别哪些代码应该自己写、哪些应该交给构造件。你知道业务规则自己写，API 入口交给框架，数据访问交给 ORM。

第三，判断实现方案中是否遗漏了胶水代码。你发现”用了 FastAPI 和 SQLAlchemy”时，能指出它们之间需要 Repository 模式的胶水代码。

产出物是结构到实现方案的决策表，标注每类结构的实现方式、具体方案和需要的胶水代码。
