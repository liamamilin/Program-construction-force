5  Chapter 5：程序表达单元：数据、集合、判断、运行、失败与反馈（Program Expression Units）

5.1 本章问题

第 4 章建立了从语法到表达的转换——你知道 for + if 是筛选、class + fields 是表示、if + raise 是验证。但这些表达是散落的。程序需要的不只是单个表达——它需要知道程序在”表示什么数据”、“处理什么集合”、“做什么判断”、“怎么推进运行”、“怎么面对失败”。这五类表达构成程序的完整表达体系。

5.2 背景：为什么这章重要

程序表达单元层（Program Expression Unit Layer）是支撑模块中的概念层。它位于基础语法之上、程序结构建模之下，帮助理解代码如何表达局部含义，但不决定程序的整体结构。

程序表达单元是基础语法组合出的、具有稳定语义的局部代码模式。它不是语法（语法只是材料），也不是结构（结构是整体组织）。它是介于两者之间的中间层——帮助你在写代码时知道自己”在表达什么含义”。

这五类表达单元是：

  1. 数据表示表达：程序如何描述现实事物
  2. 集合处理表达：程序如何处理一批数据
  3. 判断与规则表达：程序如何做决策
  4. 运行推进表达：程序如何推进状态和执行
  5. 失败与交付表达：程序如何处理失败并产出结果

注意：这五类表达是局部含义的集合。它们不等于程序结构。程序结构（Part 3）是把这些表达组织成整体规格，而不仅仅是分类。

5.3 核心内容

5.3.1 程序表达单元的定义

程序表达单元是基础语法组合出的、具有稳定语义的局部代码模式。

维度 说明
解决的问题 从”写了什么语法”到”表达了什么含义”的转换
输入 一组基础语法元素
输出 一个具有明确程序含义的局部结构
边界 不跨模块，不涉及完整业务流程，只描述局部意图
取舍 同一表达单元可以有多种语法实现

5.3.2 数据表示表达

数据表示表达（Data Representation Expression）解决的核心问题是：现实世界的一个事物如何被程序表示。

表示（Representation）：将现实对象映射到程序数据结构。

@dataclass
class Task:
    id: int
    title: str
    status: str
    priority: str

字段（Field）：对象特征在程序中的具象化。每个字段有类型和约束。

title: str        # 字符串类型
priority: str     # 枚举值：low/medium/high/urgent
due_date: date | None  # 可选日期

关系（Relation）：对象之间的关联。通过引用字段或集合字段表示。

assignee_id: int        # 引用 User 的 id
comments: list[Comment]  # 一对多关系

集合（Collection）：一组相同类型的数据单元。

tasks: list[Task]  # 任务集合

配置（Configuration）：程序的非业务参数。

MAX_TASKS_PER_PROJECT = 100
RETRY_LIMIT = 3

结果（Result）：操作返回的结构化输出。

@dataclass
class TaskResult:
    success: bool
    task: Task | None
    error: str | None

5.3.3 集合处理表达

集合处理表达（Collection Processing Expression）解决的核心问题是：如何操作一批数据。

遍历（Iteration）：逐个访问集合中的元素。

for task in tasks:
    print(task.title)

筛选（Filtering）：从集合中选出符合条件的子集。

active_tasks = [t for t in tasks if t.status == "in_progress"]

转换（Transformation）：把集合中的每个元素转换成另一种形式。

titles = [t.title for t in tasks]

查找（Lookup）：根据条件从集合中找到特定元素。

task = next((t for t in tasks if t.id == task_id), None)

分组（Grouping）：按某个键值将集合分组。

from collections import defaultdict
by_status = defaultdict(list)
for t in tasks:
    by_status[t.status].append(t)

排序（Sorting）：按规则排列集合。

sorted_tasks = sorted(tasks, key=lambda t: t.priority)

聚合（Aggregation）：从集合中计算一个汇总值。

total_tasks = len(tasks)
high_priority_count = sum(1 for t in tasks if t.priority == "high")

5.3.4 判断与规则表达

判断与规则表达（Judgment and Rule Expression）解决的核心问题是：程序如何做决策。

条件（Condition）：单个布尔表达式的真假判断。

if task.priority == "urgent":
    handle_urgent(task)

规则（Rule）：多条件组合的决策逻辑。

if task.status == "open" and task.assignee_id is not None:
    can_start = True
elif task.status == "in_progress":
    can_start = False

验证（Validation）：拒绝非法输入或状态。

if not title or len(title) > 200:
    raise ValidationError("title 长度不合法")

权限（Permission）：按角色限制操作。

if user.role not in ("admin", "manager"):
    raise PermissionError("无权操作")

限制（Restriction）：禁止某些操作的条件。

if task.status == "closed":
    raise StateError("已关闭任务不可操作")

触发条件（Trigger Condition）：事件发生后自动执行动作的条件。

if task.status == "done":
    notify_creator(task)

5.3.5 运行推进表达

运行推进表达（Execution Progression Expression）解决的核心问题是：程序如何记录状态、推进运行。

状态（State）：记录对象当前所处的阶段。

task.status = "open"

状态变化（State Change）：对象从一个阶段进入另一个阶段。

task.status = "in_progress"  # 从 open 到 in_progress

动作（Action）：封装一组相关操作。

def assign_task(task_id, user_id):
    task = get_task(task_id)
    validate_can_assign(task)
    task.assignee_id = user_id
    task.status = "in_progress"
    return task

流程（Flow）：动作按顺序编排。

def handle_create_task(request):
    validated = validate_input(request)
    task = create_task(validated)
    if task.priority == "urgent":
        assign_urgent(task)
    notify_creator(task)
    return task

5.3.6 失败与交付表达

失败与交付表达（Failure and Delivery Expression）解决的核心问题是：程序面对失败怎么处理、怎么交付结果。

失败（Failure）：程序遇到异常情况的状态。

try:
    result = execute(task)
except DatabaseError:
    task.status = "failed"
    log_error(e)

恢复（Recovery）：从失败中恢复到一致状态。

try:
    result = execute(task)
except DatabaseError as e:
    if retry_count < MAX_RETRIES:
        retry(task, retry_count + 1)
    else:
        task.status = "failed"
        log_error(e)

输出（Output）：程序把结果返回给调用者。

return TaskResponse(task=task, success=True)

反馈（Feedback）：通过日志、通知等渠道告知用户结果。

logging.info(f"Task {task.id} created by {user.name}")
notify_assignee(task)

5.4 例子与实践

5.4.1 场景：Task Manager 的表达单元标注

以”完成任务”功能为例，标注每段代码涉及的表达单元：

# 数据表示：Task 对象定义
@dataclass
class Task:
    id: int
    title: str
    status: str
    assignee_id: int | None

# 判断与规则：验证是否可以完成任务
def validate_can_complete(task: Task, user_id: int):
    if task.status != "in_progress":         # 条件表达
        raise StateError("任务不在执行中")     # 验证表达
    if task.assignee_id != user_id:           # 条件表达
        raise PermissionError("不是负责人")    # 权限表达

# 数据表示：请求和响应结构
@dataclass
class CompleteTaskRequest:
    task_id: int
    completion_note: str

@dataclass
class TaskResponse:
    success: bool
    task: Task | None
    error: str | None

# 运行推进：完成任务的动作
def complete_task(request: CompleteTaskRequest, user_id: int) -> TaskResponse:
    task = get_task(request.task_id)                  # 查找表达
    validate_can_complete(task, user_id)               # 验证表达
    task.status = "done"                               # 状态变化表达
    save_task(task)                                    # 外部交互
    notify_creator(task)                               # 反馈表达
    return TaskResponse(success=True, task=task)        # 输出表达

5.5 小结

程序表达单元层是介于基础语法和程序结构之间的中间层。五类表达单元（数据表示、集合处理、判断与规则、运行推进、失败与交付）覆盖了程序的所有局部含义。

但表达单元只是局部含义。它不回答程序是什么——一个筛选表达可以出现在记账程序、游戏程序、聊天程序中。表达单元是零件，不是蓝图。

下一章进入 Part 3：程序结构建模层——把需求中的稳定结构抽取出来，而不是把表达单元升级成更大的代码说明。

5.7 从业者输出

完成本章后，你能做三件事：

第一，对任意一段代码，标出它包含了哪些表达单元。你能同时说出”这段代码用了什么语法”和”这段代码表达了什么”——两个不同层次的理解。

第二，当你读别人代码时遇到不理解的片段，你会用”这个片段在表达什么”来追问——而不是”这个语法怎么写”。

第三，区分”表达单元”和”程序结构”。你知道一个筛选表达是局部含义，而筛选在”用户管理系统”中服务于”查找活跃用户”这个整体结构。
